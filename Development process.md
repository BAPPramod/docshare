4	Development Process
This chapter walks through how I actually built DocShare from start to finish. I cover the decisions I made during implementation, the structure I ended up with, how the different parts of the codebase connect to each other, and the problems I ran into along the way. Building a full-stack application alone over a few months means there are always moments where things do not work the way you expected, and I think those moments are worth talking about because they shaped what the final product looks like.
4.1	Planning and Project Setup
Before writing any code, I spent time thinking through the feature set and what the folder structure should look like. I wanted to keep the scope realistic and well-defined. The core requirement was simple: users should be able to register, log in, upload documents, and share them with specific other users. Everything else, like audit logging, file type validation, and ownership-based delete permissions, was a security and quality concern that I layered on top once the basics were working.
I set up the project as a monorepo with a frontend folder and a backend folder sitting under the same root directory. Both sides use TypeScript, and having a root tsconfig.json that uses project references to point at the backend and frontend configurations separately was a clean way to handle this. Each side keeps full control over its own compiler settings, but they can still be built together from the root when needed. It also made the workspace feel properly connected rather than just two unrelated projects sitting in the same folder. Having one Git repository for the whole application kept version control straightforward. I could commit a frontend and backend change together when they were related, which made the commit history easier to follow.
I also created a .env.example file in both the frontend and backend folders to document which environment variables are required. This meant that if I came back to the project after a break, or if someone else wanted to run it, the setup requirements were clear without needing to dig through the code to find out what configuration was expected.
For the backend I initialised a Node.js project and installed the key runtime dependencies: Express.js for the HTTP server, Prisma for database access, jsonwebtoken for creating and verifying JWTs, bcryptjs for password hashing, Multer for handling file uploads, and cors for cross-origin request control. I used tsx in watch mode for development so that TypeScript files could be executed directly and the server would restart automatically whenever I saved a change. For the frontend I used Vite to scaffold a Vue 3 TypeScript project and added Tailwind CSS, Pinia, TanStack Query, and Vue Router on top.
4.2	Backend Structure and Entry Points
The backend is split across two entry files. server.ts is the process entry point and does almost nothing except import the Express app from app.ts and call app.listen on the port read from the PORT environment variable, defaulting to 3000 if PORT is not set. Keeping it this thin was a deliberate choice because it means tests can import app.ts directly and exercise the routes without starting a real server process.


app.ts is where the Express application is actually assembled. It applies the CORS middleware using the default configuration, registers express.json() for parsing JSON request bodies, mounts a static file handler that serves uploaded files from the uploads directory, registers the three route modules at their base paths (/api/auth, /api/documents, /api/shares), and adds a /health endpoint that returns a JSON response with a status field and a timestamp. That health endpoint turned out to be useful during development for quickly confirming the server was alive and responding.


The three route files are intentionally thin. Each one maps URL paths to the right combination of middleware and controller function and nothing else. Business logic stays in the controllers. The documents router, for example, applies the authenticate middleware once at the router level using router.use(authenticate), which means every route defined after that line is automatically protected. The individual route definitions then only specify the path and the handler, like this:


router.post("/", upload.single("file"), uploadDocument);
router.get("/", getDocuments);
router.get("/:id/download", downloadDocument);
router.delete("/:id", deleteDocument);


Having authenticate applied at the router level rather than repeated on every single route made the file cleaner and removed the risk of accidentally leaving a route unprotected. The shares router follows the same pattern. The auth router does not use the authenticate middleware at all because its endpoints handle registration and login, which by definition happen before a user has a token.


4.3	Authentication Implementation
Authentication was the first feature I built because everything else depends on knowing who is making a request. The auth controller in src/controllers/authController.ts handles register and login.
For registration, the controller reads the email, password, and name from the request body, confirms all three are present, queries the database to check the email is not already taken, hashes the password with bcryptjs using 10 salt rounds, and creates the User record. I went with 10 rounds because it felt like a reasonable middle ground between security and response time for this kind of application. Going higher would slow down every registration and login noticeably, and for a project at this scale 10 rounds already makes brute-force attacks impractical.
For login, the controller finds the user by email, runs bcrypt.compare against the stored hash, and if the credentials match, signs a JWT containing the user's ID and email with a 24-hour expiry. The response for both registration and login returns the token alongside a safe user object containing the id, email, and name. The password hash is never included in any response. That exclusion is enforced in the controller by explicitly selecting which fields to return, not just assumed by the frontend.
The JWT utility lives in src/utils/jwt.ts. It checks at startup that JWT_SECRET is set in the environment and throws immediately if it is not, rather than failing silently later. The generateToken and verifyToken functions are both exported from this single file so the auth controller and the auth middleware import from the same place, keeping the token structure consistent across the application.
The auth middleware in src/middleware/auth.ts reads the Authorization header, confirms it starts with "Bearer ", extracts the token string, and calls verifyToken. Any failure at any step returns a 401 and stops the request from going further. On success it attaches the decoded payload to req.user so every downstream controller can read the user's ID and email without making another database call.
4.4	Document Handling
The document controller in src/controllers/documentController.ts handles four operations: upload, list, download, and delete. Each has its own authorization logic because the access rules differ between them.
Upload goes through the Multer middleware defined in src/middleware/upload.ts before it reaches the controller. Multer is configured with a diskStorage engine that saves files into the uploads folder. The filename is generated by combining Date.now() with a large random integer from Math.round(Math.random() * 1E9), then appending the original file extension. This produces names like 1704567890123-394857123.pdf, which avoids collisions without needing a UUID library. The fileFilter function checks the file's MIME type against the allowlist of application/pdf, image/jpeg, and image/png and rejects anything else. The maximum file size is capped at 10 MB. Because Multer runs before the controller, by the time the controller code executes the file has already been validated and written to disk. The controller then writes the Document record to the database, storing the generated filename, the original filename the user uploaded, the MIME type, the file size in bytes, the disk path, and the owner's ID taken from req.user.
Listing documents returns two separate arrays in a single response: documents the user owns and documents that have been shared with them. Both queries go through the Document model in Prisma. The owned set is filtered by ownerId. The shared set is fetched by filtering documents where the shares relation has at least one entry matching the requesting user's ID, using Prisma's some relation filter. Both queries include the owner and shares data so the frontend has everything it needs to render the document cards without making additional requests.
Download is the most security-sensitive operation because it is where unauthorized access would actually expose a file. The controller runs a single Prisma query that looks for a document matching the requested ID where the user is either the owner or has a share entry. If no matching document is found, meaning the document does not exist or the user does not have access, the controller returns a 404. I went with 404 rather than a separate 403 because I did not want the endpoint to reveal whether a document ID exists to someone who should not know about it. If access is confirmed, the controller creates an AuditLog entry recording the download action and then calls res.download with the resolved file path and the original filename. Passing the original filename is important because without it the browser would prompt the user to save a file with the randomly generated internal name, which is meaningless to them.
Delete is restricted to the document owner. The controller queries for a document matching both the requested ID and the requesting user's ID as the owner. If no match is found it returns a 404. If ownership is confirmed, the controller first checks whether the physical file still exists on disk using fs.existsSync and removes it with fs.unlinkSync if it does. After the file is removed from disk, it then calls prisma.document.delete, which cascades to the related DocumentShare and AuditLog rows based on the onDelete: Cascade rules in the schema. Doing the file removal before the database delete means that if the unlink fails for some reason, the database record is still intact and a retry is possible, which felt like the safer ordering.
4.5	Sharing Logic
The share controller in src/controllers/shareController.ts handles three operations: sharing a document, removing a share, and searching for users.
To share a document, the controller first checks that the requesting user owns it. This is a database check, not a frontend assumption. It then looks up the target user by email address. If no user with that email exists, it returns a 404. It also rejects any attempt to share a document with yourself, returning a 400. Before creating the share record, it checks the DocumentShare table for an existing row with the same documentId and userId combination. If one already exists, it returns a 400 rather than silently creating a duplicate. The Prisma schema also has a unique constraint on that combination as a second line of defence. If all the checks pass, it creates the DocumentShare record and returns the new share with the target user's details included.
To remove a share, the controller confirms ownership of the document again, then deletes the matching DocumentShare row using the composite documentId and userId key.
The user search endpoint returns registered users filtered by an optional search query string, excluding the requesting user from the results and limiting the response to 10 entries. This endpoint exists on the backend and is available for the frontend to use, though the current ShareModal component takes a simpler approach by letting the user type an email address directly rather than searching through a user list.
4.6	Frontend Structure
The frontend entry point is src/main.ts. It creates the Vue application instance, registers Pinia first, then the Vue Router, then the TanStack Query plugin, and finally mounts the app to the #app element in index.html. Plugin registration order matters here because both the router and the query client get injected into the Vue app context and need to be available before any component tries to use them.
The app shell is App.vue, which is kept minimal on purpose. It renders nothing except a router-view, which means the currently active route component fills the entire page. The Layout component, which provides the navigation bar, the logout button, and the main content wrapper, is used inside the Dashboard page rather than in App.vue. This was a conscious choice because the login and register pages do not need the navigation bar, so wrapping them in the layout would have meant adding conditional rendering logic that felt unnecessary. Instead, Dashboard.vue simply wraps its content inside the Layout component, and the auth pages render their own standalone designs.
Pages and components live in separate folders. The pages folder has the top-level views that the router loads: Login.vue, Register.vue, and Dashboard.vue. The components folder has the reusable pieces: DocumentUpload.vue, DocumentCard.vue, ShareModal.vue, and Layout.vue. Keeping them separate makes it immediately clear what is a route destination versus what is a reusable building block.
Tailwind CSS handles all the styling. The global stylesheet in src/style.css defines a small set of component classes like .btn, .input, and .card using Tailwind's @layer components directive. These act as shorthand for commonly repeated utility combinations, which kept the individual component templates cleaner and reduced the amount of repeated class strings across the project.
4.7	Auth Store and Session Persistence
All authentication state lives in the Pinia store defined at src/stores/auth.ts using the Composition API style. The store holds the JWT token, the current user object, and a computed isAuthenticated property. It exposes three methods: login, register, and logout.
The login method calls the backend auth endpoint via the shared axios instance, saves the returned token and user object to the Pinia reactive state, writes both to localStorage for persistence across page refreshes, and sets the Authorization header on the axios instance so every subsequent API request automatically includes the token. That last step means individual API calls do not need to think about authentication headers at all. The register method works the same way.
Session restoration happens inline when the store is first created, not through a separate method. The store's setup function checks whether a token already exists in localStorage. If it does, it sets the Authorization header on the axios instance and tries to parse the stored user JSON from localStorage back into the reactive state. This runs automatically the first time useAuthStore() is called, which happens early in the app lifecycle through the router guard. There is no explicit initialize call in main.ts. It just works because Pinia evaluates the setup function on first use.
Logout clears the token and user from both the Pinia state and localStorage and removes the Authorization header from the axios instance. The router guard then handles the redirect to the login page.
The router guard in src/router.ts checks route metadata before every navigation. Routes with requiresAuth redirect to /login if the user is not authenticated. Routes with requiresGuest redirect to the dashboard if the user is already logged in. Having all this in the router file means there is exactly one place to look and one place to change when authentication routing behaviour needs adjusting.
4.8	API Layer and Data Fetching
All HTTP communication is handled through a single axios instance defined in src/services/api.ts. The base URL is set to /api, and in development the Vite dev server proxies requests from /api to [http://localhost:3000](http://localhost:3000), which means the frontend and backend appear to be on the same origin from the browser's point of view. This file also defines TypeScript interfaces for the API response shapes, so the rest of the frontend works with typed data throughout.
Two interceptors are attached to the axios instance. The request interceptor reads the current token from the auth store and attaches it as a Bearer token in the Authorization header before each request goes out. The response interceptor watches for 401 responses and calls the auth store's logout method when it sees one, clearing the session. This handles token expiry mid-session transparently, without any individual component needing to check for it.
TanStack Query wraps the API calls in the components that use them. The Dashboard page defines a useQuery with the cache key ["documents"] that calls documentsAPI.getAll(). The query is configured with refetchOnWindowFocus set to false, so the document list does not automatically refetch every time the user switches back to the browser tab. I made that choice because the constant refetching felt unnecessary for this kind of application and could be distracting if the list visibly flickered on every tab switch.
All write operations use useMutation. Each mutation's onSuccess callback invalidates the documents query by calling queryClient.invalidateQueries with the query key, which tells TanStack Query to discard the cached document list and fetch a fresh copy from the server. The UI reflects the updated state after any upload, share, unshare, or delete action this way.
Handling expired tokens relies on the axios response interceptor described above. When a 401 comes back, the interceptor calls logout and the session ends. TanStack Query's default retry behaviour may still attempt a few retries before the interceptor fires on each one, but in practice the logout happens quickly enough that users do not notice a significant delay.
4.9	Key UI Components
DocumentUpload.vue handles file selection and the upload flow. It uses a file input element with the accept attribute restricted to .pdf, .jpg, .jpeg, and .png. When a file is selected, it stores it in a reactive ref. When the user clicks the upload button, it passes the file to the upload mutation, which wraps it in a FormData object with just the file attached and sends it to the backend. On success, the mutation invalidates the documents query so the list refreshes, clears the selected file ref, and resets the file input element's value to an empty string. Resetting the input is important because without it the browser does not fire a change event if the user selects the same file again immediately after, which would make re-uploading the same file impossible without choosing a different file first.
DocumentCard.vue renders a single document with the actions available to the current user. It shows a generic document icon alongside the filename, file size, and upload date. The download button triggers a mutation that calls the download endpoint with responseType set to blob. When the response comes back, the component creates a Blob from the response data, generates a temporary object URL with URL.createObjectURL, creates a hidden anchor element with the download attribute set to the original filename, appends it to the document body, programmatically clicks it, removes it, and then revokes the object URL to release the memory. This is the standard way to trigger a browser file download from an API response in a single-page application. The share and delete buttons only appear if the current user is the document owner, which is determined by comparing the auth store's user ID against the document's owner.id property from the included relation data. The backend enforces the same ownership rules independently regardless of what the UI shows.
ShareModal.vue manages the sharing flow for a selected document. It displays a list of users the document is currently shared with and provides an email input field where the owner can type the email address of the person they want to share with. When the share button is clicked, it sends the email to the share endpoint. On success, it appends the newly shared user to a local reactive copy of the shares list so the UI updates instantly without waiting for a network round trip. It also invalidates the documents query in the background to sync the cache with the server. The unshare flow works similarly: clicking remove on a shared user triggers the unshare mutation, updates the local list immediately, and invalidates the query. The local reactive state and the background query invalidation together give the user instant feedback while ensuring the data eventually converges with what the server has.
4.10	Implementation Challenges
Building DocShare alone meant that when something broke, I had to work through it without anyone to ask. Several things came up during development that I had not planned for.
The first was CORS. When I first ran the frontend and backend at the same time, API calls from the Vue app running on port 5173 to the Express server on port 3000 failed with CORS errors in the browser console. The browser was treating the two ports as different origins and blocking cross-origin requests. The solution I settled on was configuring a proxy in the Vite dev server so that requests to /api get forwarded to [http://localhost:3000](http://localhost:3000). This way the browser sees everything as coming from the same origin and CORS is not an issue during development. On the backend side I still have the cors middleware installed with default settings as a fallback for any scenario where the proxy is not in the picture, but in day-to-day development the proxy is what actually makes things work.
The second challenge was triggering file downloads in the browser. When I first added the download button and tested it, clicking it opened a blank tab instead of downloading anything. The frontend was navigating to the download URL rather than fetching the response as a file. The fix was to use axios with responseType set to blob, take the blob from the response, create an object URL with URL.createObjectURL, attach it to a dynamically created anchor element with the download attribute set, programmatically click the anchor, and call URL.revokeObjectURL afterwards. This is a well-known pattern for SPA downloads but it was not something I had done before and it took some reading to find.
The third challenge was TypeScript and Multer. Multer adds a file property to the Express request object after processing an upload, but the standard @types/express Request type does not include that property. TypeScript flagged a type error on every line that accessed req.file. The fix was to install @types/multer, which augments the Express Request type with the file and files properties. Once that package was in the devDependencies, TypeScript recognised req.file and the errors went away.
There was also a subtle issue with the delete endpoint that I only caught during end-to-end testing. My first implementation deleted the database record but left the physical file sitting in the uploads folder on disk. The fix was straightforward: before calling prisma.document.delete, I added a check with fs.existsSync and a call to fs.unlinkSync to remove the file from disk. I put the file removal before the database delete intentionally, because if the unlink were to fail, I wanted the database record to still be intact so the operation could be retried. If the file happens to already be missing for whatever reason, the existsSync check means the delete still proceeds normally without throwing an error.
